## bc划分的原因
当bc过大时的问题：
1. 模型内部的概念是需要统一的，一个概念不能有多种解释
2. 在大型系统中拉齐模型认知的代价非常高，甚至不可能
3. 兼顾所有人的需求会导致模型本身不够精简
4. 权力和组织架构导致模型天然就会分裂

当bc过小时的问题：
难以保持集成度和一致性

## bc存在的意义
bc定义了每个模型的应为范围，并利用context map来定义bc内部和外部的关系
相当于bc本身内部对模型达成一致，当对外交流时，需要使用context map作为adapter来进行适配和转换

感觉实际的bc划分，基本是以一个小组完成的一个子系统来定义的，例如订单系统，数据库系统。领域模型产生过程是一个反复讨论迭代的过程，这基本限制了它很难在一个人数很多的群体中展开。
不同组之间对模型的理解基本一定会是有偏差的

使用其它团队的数据库表单是一个典型的问题，因为schema也是bc的一部分，这样相当于侵入对方的bc

## 识别bc内的不一致现象
1. 重复概念
两个模型代表了同一个概念，所以每次都需要同步发生变更
2. 假同源
使用相同术语的人以为他们谈论的是同一个概念，但实际上不是

解决bc内不一致的手段：
### 持续集成
1. 把一个上下文中的工作足够频繁的拼合到一起
2. 当模型分裂时可以迅速的发现
3. 两个级别：模型概念的集成，实现的集成

## context map
1. bc之间的代码、schema重用有很高的风险，功能和数据的集成需要通过转换来实现
2. context map位于各个bc间的重叠的部分

context map就是一种bc子系统级别的适配器

## bc之间的合作模式
### shared kernel
1. 多个bc的团队共享一部分代码内容，可能时某部分基础层，或者一些核心领域概念
2. 这部分的修改频率和集成频率都比较低，变更要所有bc的知晓和测试
3. 目标是减少重复

感觉这种模式再宏观一些就是中台模式，大家共享一些稳定且基础的服务

### 客户/供应商模式
开发中常见的模式，一个系统向它依赖的各个子系统提出需求订单，作为客户。各个依赖方完成订单需求，作为供应商。

需要有完整的测试套件来快速识别供应商变化的影响

必要的条件：组织架构上，其它系统愿意以供应商的模式进行合作

### separate way
当前bc缺乏对其它bc的约束力，但是对其它bc有强依赖

两种情况：
1. 上游的设计非常难用，那么自己需要完全设计复杂转换层，对应防腐层方式
2. 上游的设计很好且兼容，这时采用跟随者模式，严格遵守上游团队的模型，不转化

#### 跟随者模式CONFORMIST
好处：集成非常简单，不需要对模型进行转换和设计
坏处：对上游的依赖非常深

类似于shared core，但是可以在完全没有合作的情况下进行，是一种跟随和模仿

#### ANTICORRUPTION LAYER防腐层
防腐层会在bc的内部和外部之间进行双向转换，用来保护bc自己的模型

#### open host service
感觉是建立一个对外暴露的公共网关，外部统一使用这个统一的网关来接触bc，只需要维护这个网关，代价比较低
