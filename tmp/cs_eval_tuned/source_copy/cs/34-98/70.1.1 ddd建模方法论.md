# 为什么需要建模

  

模型的三个基本用途[eric]

1.  模型和设计的核心互相影响
    
2.  模型是团队所有成员使用的通用语言的中枢
    
3.  模型是浓缩的知识
    

  

抽象的知识对个体的意义可能有限，但是它一定可以更有效的帮助记录、分享和传播，人类的知识传播体系都是基于抽象的。抽象化的知识可以带来很多明显的好处：

  

1.  开发团队内部、开发和业务专家之间达成共识，降低沟通成本
    
2.  代码的接手成本更低，只要模型和架构清晰，代码和实现的细节可以是『lazy load』的
    
3.  一些设计模型的好处，低耦合、ocp等等，降低变更成本
    
4.  增加成就感，获得对真实世界业务的认知
    

  

另外一点，建模是一个非常明确的动作，而且大家非常熟悉，非常适合作为DDD的启动方式，可以很大程度的降低初期实施的阻力。

# 什么是好的模型

判断好模型的几个标准

1.  反映真实业务场景，容易被其它角色认可，有沉淀的价值
    
2.  模型泛化能力强，可以满足未来很大范围的变化
    
3.  提炼、反映了很多隐式的规则，大家对业务的理解在建模的过程中被强化了
    

  

模型的一个前提是它需要能够反映真实业务场景，在这样的前提下我们才可以获得上述的好处，一个自己觉得满意，但是和真实世界脱钩的模型是缺乏价值的。模型是存在相对的grand truth的，越靠近这个gd，大家就越容易达成一致，这种达成一致理解的能力可以作为是否有效的一个指标。

  

泛化能力是建模的核心之一，如果每次发生需求变更都需要对模型进行重新的调整，那很难说这是一个好的模型，也说明之前的建模并没有有效的反映出这个业务场景下的需求，知识简单的对业务需求的跟随。

提升泛化能力的手段之一是要避免闭门造车，和其它的研发团队、领域专家进行更多的讨论和交流，同时提升自己的产品视野，对业务本生的变化有自己的判断，并尝试在建模的过程中实验模型对这些变化的适应性。

  

隐式规则的产生代表模型进入了全新的境界，建模这个过程已经有效的推动了对未知业务知识的了解，具体参考后续c9的内容

  

另外一个对编码的要求，模型的设计和代码的实现需要有良好的对应关系，从变量命名、代码结构上均可以正确的反映模型，如果需要依赖很多文档来表达模型，那模型的精确性、维护成本都不会太好。

  

# 建模方法论

存在一些系统化的建模方法，比如『事件风暴』，这里不展开，简单讲讲一般建模中的一些思路。

  

白板是建模的核心工具。建模的过程其实是假设+验证的过程。

首先可以先提出一个粗糙的基础模型，大家围绕这个模型进行一些讨论：

1.  这个模型解决了什么问题，带来了什么好处
    
2.  这个模型是真实业务的反映吗
    
3.  举反例，这个模型不能兼容哪些情况
    
4.  讨论模型的泛化性，如果我们未来这样变化模型是不是就失效了
    

  

建模的过程需要产出一个粗糙的UML图，作为讨论的结果，UML图需要被记录，作为下一次讨论的开始点。

  

模型的细化是一个反复迭代的过程，每次讨论完成，大家需要对本次讨论的内容进行独立的思考，反复思考上述的问题，并且在下次讨论时给出自己的问题和方案。反复进行这个过程，直到到达一个大家都基本满意的结果。
