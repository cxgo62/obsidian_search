## 概览
1. 表锁一般不会出现死锁，多个表锁会一次性原子的被申请
2. 死锁问题一般出现在行锁
3. innodb会利用DFS搜索进行环检测来防止死锁，如果有环或者深度超过200会选择一个代价较低的事务进行回滚

## 死锁的例子
```sql
CREATE TABLE `student` (
`id` int NOT NULL,
`uuid` varchar(64) NOT NULL,
`name` varchar(64) NOT NULL,
`age` int NOT NULL,
PRIMARY KEY (`id`),
UNIQUE KEY `uuid_index` (`uuid`),
KEY `name_index` (`name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci
```

假设没有id=100的记录，RR级别
![[Pasted image 20211012113442.png]]
select 加一个间隙锁来防止幻读，insert会加插入意向锁，这两个锁会冲突

- 没写业务需要提前锁住间隙，防止并发插入同样的数据，导致先`SELECT...FOR UPDATE`再`INSERT`，这样在高并发情况下很容易死锁
- 最好直接利用唯一索引来实现同样的目的

### 4 唯一键时，一个事务回滚导致其它事务死锁
假设uuid=uuid100不存在
![[Pasted image 20211012114144.png]]
1. 三个事务同时插入一条某个唯一索引属性（上面的uuid）相同的数据
2. 其中某个事务先一步插入，其他两个事务会阻塞等待
3. 然后先一步插入的事务回滚
4. 其他两个事务出现死锁，其中某个事务会被回滚

- `INSERT`的时候，会对唯一索引进行Duplicate Key判断，如果唯一键冲突，则会加共享锁等待，也就是T3时候的事务二和事务三，都会获得共享锁。
- T4时，事务一回滚，事务二和事务三都会申请==升级排他锁==，这样就造成类似案例2的死锁的情况

### 5 高并发下聚簇索引和二级索引的加锁顺序
![[Pasted image 20211012141820.png]]
1. 事务1的加锁顺序是先锁二级索引name_index，再锁聚簇索引
2. 事务2的加锁顺序是先锁聚簇索引，再锁二级索引
3. 不同的加锁顺序可能在==高并发==情况下导致死锁

## 避免死锁的tips
简单来说，几个关键点
1. 不同的查询用相同的顺序访问表和行
2. 经量使用小事务，拆解大事务
3. 尽量合理的索引，==在RR下，非索引查询将会为表的每一行记录加锁==
4. 少用for share或者for update，由于间隙锁和临键锁的存在，锁住的可能不止是一行记录
5. 提前申请足够强度的锁，==不要先用for share锁住行，后面再update，很容易死锁==
