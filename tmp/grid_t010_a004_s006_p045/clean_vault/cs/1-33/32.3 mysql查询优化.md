## 优化数据访问
分析的步骤
1. 用户是否在检索大量超过需要的数据，也就是访问了太多的行或者太多列
2. mysql的服务层是否在分析大量超过需要的数据行

### 请求不需要的数据
1. 多表关联时返回全部列
2. 使用select *

### 扫描额外的数据
衡量查询开销的三个指标
1. 响应的时间
2. 扫描的行数
3. 返回的行数

响应的时间 = 服务时间 + 排队时间
最理想情况下， 扫面的行数 == 返回的行数

where语句的三种方式，从好到坏
1. 在索引中通过条件来过滤不匹配的记录，在引擎层完成
2. 使用索引覆盖扫描（use index），直接从索引中过滤不匹配的记录并返回，但这是在服务层进行的，没有回表查询
3. 在数据表中返回不需要的记录，过滤是在服务层完成的

## 重构查询方式
### 查询切分
1. 不要使用特别重的查询，可以对查询进行差分，分多次进行
2. 一个典型的例子是大规模的删除数据可能导致锁的时间非常长，可以切分成小的操作

### 少使用join
将join的操作分多次用in来替代

## 查询执行的基础
### 状态查询
命令：SHOW FULL PROCESSLIST 查看线程状态
1. sleep 空闲等待
2. query 正在执行查询或者正在发送结果
3. locked 在等待服务器层的表锁，引擎层的锁不会体现在线程状态中
4. analyzing and statistics 搜集引擎的统计信息，并生成执行计划
5. copying to tmp table 一般在做group by、文件排序、union
6. sorting result 对结果集排序
7. sending data 在生成结果集或者传输数据

### 查询缓存
缓存通过一个大小写敏感的hash查找实现

### 查询优化
先通过解析器将sql语句解析威解析树，并验证语法的合法性

优化器
1. 基于成本考虑
2. 尝试预测某几种查询计划的成本，并选择成本最小的一种
3. 成本的最小单位是随机读取一个4K数据页的成本
4. Last_query_cost可以获取查询成本

优化器选择错误的场景
1. 引擎提供的统计数据不准确
2. 估算的算法本身的局限性，估算是以磁盘io作为标准的，但是没办法考虑内存和cpu，也没办法考虑cache等因素
3. 优化是基于成本而非速度，所以可能不会给出延时最低的方式
4. 优化不考虑并发的情况

优化的分类
1. 静态优化：变化查询到等价的另外一种形式
2. 动态优化：根据统计信息动态调整

优化器的静态优化  [](marginnote3app://note/6D47F016-9370-4FA1-98C4-7DA13F944DFA)
