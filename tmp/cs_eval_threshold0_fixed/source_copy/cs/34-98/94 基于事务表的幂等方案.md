## 概要
在很多场景下需要处理请求方多次重复调用的问题，保证业务接口幂等

有些场景是天然幂等的，比如将一个订单的状态设置为DONE，反复调用并无影响

但在一些类似于累加的场景中，因为存在inc操作，必须要额外的手段来保证只inc一次


## 方案1
在请求到达前进行过滤，添加一个拦截层
### 内存方案
1. 对特别的业务ID（如订单ID）进行hash，确保ID一样的请求只会落在一台机器上
2. 当请求到达后，在单机内存中记录ID，并在每次请求时判断ID是否已经在内存中存在了
3. 如果执行失败，则清除内存中的ID
   
### redis方案
对于无法实现hash分发的请求，可以利用redis替代单机内存来实现过滤

redis的操作必须考虑并发的冲突，所以需要使用SETNX或GETSET等原子命令，利用redis自身的原子性来变相实现一个分布式锁

### 问题
调用链路大致如下：
```mermaid
graph LR;
	request --> filter --> 业务逻辑
```
1. 如果在filter记录成功后，进程中断，则业务逻辑永远没有办法被执行
2. 业务逻辑执行失败时同理，如果filter回滚之间进程中断，将丢失这次操作

本质的原因：filter的标记和业务逻辑的执行不在一个事务中，这是一个两方事务

## 方案2
在不考虑分布式事务的方案下，单利用数据库的local 事务来解决，除开业务自己的表，单独维护一个事务状态表xid_status，记录（XID，status）这样的信息

1. 利用请求自身的唯一性实现一个XID
2. 不使用任何filter，直接进行到业务逻辑事务操作阶段
3. 业务逻辑的local 事务中包含几个阶段：
	1. 利用select for update锁定这个xid_status的一个row
	2. 判断这个XID的状态是否为done，如果为done则事务回滚，否则继续
	3. 执行业务DML
	4. 将 XID的状态设置为Done
	5. commit

通过添加这样的独立的事务表，可以确保每个XID对应的请求只可能被执行一次，配合调用发起方的重试策略，可以实现exactly once

缺点：性能较差，普通写请求都变成了事务写
