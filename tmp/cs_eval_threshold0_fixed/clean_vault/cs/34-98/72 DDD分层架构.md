# 分层原则

分层架构的本质：关注点分离

> 要想创建出能够处理复杂任务的程序, 需要做到关注点分离——使设计中的每个部分都得到单独的关注。在分离的同时,也需要维持系统内部复杂的交互关系。

  

开发人员不应该在写业务逻辑的时候关心网络丢包的问题，同理，在实现领域层的时候也不需要关注数据的物理存储，分层的意义就是彻底剥离这种关注的负担。

  

分层的基本原则：

1.  层中的任何元素都仅依赖于本层的其它元素或者下层的元素
    
2.  向上的通信必须依赖间接的方式进行（消息、事件等）
    

  

# 对称架构

标准的四层架构

interface->application->domain->infrastructure

  

# 非对称架构

六边形架构更好的阐述了核心业务逻辑的概念，也更符合DDD的核心哲学思想。

在非对称架构中，整个软件被分位了内部和外部。这种定义明确的提示了开发人员的注意力分配方式。

内部：

1.  位于内部的逻辑代表了整个业务的本质，需要投入最多的精力和资源
    
2.  是相对稳定且容易被理解的
    
3.  不应该存在研发不理解的黑盒
    

外部：

1.  和外部的交互都通过不同的接口适配器实现，它们只是信息的流入流出
    
2.  很容易被替换，对外部的了解程度不应该阻碍对业务的理解
    
3.  外部的信息格式是多种多样的，需要通过适配器翻译成为一个对内部统一的格式
    

  

实际的开发过程中，可以粗糙的将内部等同于应用层+领域层，将外部等同于接口层和应用层

  

## 如何理解接口适配器模式

设计模式中，适配器是用来进行不同组件之间的语义转化，比如：

1.  对不同存储后端的sql语句实现的封装
    
2.  对输入（rpc/http）的参数提取
    

  

这些都是协议级别的适配，屏蔽了不同的协议、不同的数据库语句之间的差别，让内部逻辑可以更加标准化的对不同的外部信息进行存取

  

个人理解，DDD中的接口适配器模式除了上诉的协议级别的装换之外，可以在语义级别进行不同BC之间的通用语言的转换，也就是实现了context map的能力。

每个系统的『内部』是工作在自己的通用语言和建模中的，和外部的信息存取很可能是无法兼容的，这就要求有一个可以理解双方的通用语言的组件，来起到翻译的作用，防止内部去尝试理解外部逻辑导致的问题

  

# 分层架构的副作用

分层架构完全是从业务逻辑、业务建模的角度考虑的，这往往让它们在运行时存在很大的性能问题。

性能问题不是来自额外的抽象和函数调用，更多的时因为在建模时对底层基建缺乏考虑，导致数据库等组件无法以高效的方式被使用。

  

举一个例子，做一个批量修改用户功能，理想的应用层的伪代码可能是这样

```
users = userRepo.getAllUsers()
  
users.each().setNewInfo(newInfo)
  
users.each().save()
```

  

这个很符合业务逻辑，也是很标准的应用层写法，但确有严重的性能问题，更多的我们会写成

```
userRepo.updateAllUserInfo(newInfo)
```

  

这种方式让业务逻辑泄露到了基建层(适配器)中，但确是最可行的实现方式，所以很多时候理想的分层架构是无法真正工作的。

  

分层架构还会在跨repo事物等方面遇到一些更复杂的挑战，这里先不展开。总体来说对分层架构的设计是无法做到完全的隔离和透明的，还是需要根据实际情况去调整和妥协。
