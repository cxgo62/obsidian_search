## 概述
利用kafka等MQ可以实现基本的事件驱动架构
生产方生产事件，消费方维护一个worker不断轮训MQ，是一种pull模式

pull模式有很多的问题：
1. 可能存在多套消费的worker，这些worker的管理会非常麻烦，也不可靠
2. 业务开发人员需要单独支持这种pull模式，比较麻烦，可能重复开发

解决的思路之一就是==把pull变成push==

开发一个通用的消息总线组件，这个组件完成的功能：
1. 用户注册一个MQ和一个回调地址
2. 消息总线会读取这个MQ，并讲消息转发到注册的回调地址
3. 实现队列和回调的统一管理，例如积压报警、消息缺失报警、回调失败报警，并提供监控大盘

基于这样的消息总线，开发者可以用很低的代价接入一个异步事件源：
1. 开发一个接口，用于处理某个特别的事件。这个接口可能原本就已经存在了
2. 在消息总线上注册MQ的地址和回调地址
3. 在消息总线上设定匹配和数据格式转化规则

利用这样的方式，开发人员可以忽略异步和同步的数据来源，以一种统一的方式进行开发，也不用对MQ进行管理

事件总线是serverless的基建，可以比较方便的实现编排 [[92.3 seata SAGA#编排（Choreography）]]

## 实现事件总线的方式
1. 发布事件到消息队列，消费事件方进行订阅
2. 利用maxwell等对数据表进行增量同步，并在同步的过程中对数据进行过滤和转换

## 阿里云的事件总线EventBridge
兼容云原生的业界的标准  CloudEvents
### 事件 event
状态变化的数据记录

### 事件源 Event Source
事件的来源，生产者
1. 数据库变更
2. 31 阿里云 DTS
3. 消息中间件

### 事件总线 Event Bus
接收事件源的事件

### 事件规则 Event rule
1. 事件的过滤
2. 事件的转换
3. 事件的目标

类似于ETL，对事件进行筛选后，变换为特定的格式，发送给相应的上游服务

#### 事件目标
1. 常见的事件目标是http/https的服务，需要在事件总线中提前进行注册
2. 其它的包含一些信息推送服务（邮件、push）
3. severless下目标可能是一个函数计算FC

### 连接器
从特定的事件源拉取数据，推送到数据总线
