## 概览
- hyperloglog（hll）是一种基数估计法，通过概率预估一个不重复元素，也就是一个set的数量
- 可以用于统计pv uv等
- 基于set实现的这种计数的内存增长是O(n)的，不符合大数据量的场景

目前用于基数计数的概率算法包括:  
  
> -   **_Linear Counting(LC)_**：早期的基数估计算法，LC在空间复杂度方面并不算优秀，实际上LC的空间复杂度与简单bitmap方法是一样的（但是有个常数项级别的降低），都是O(Nmax)；
> - **_LogLog Counting(LLC)_**：LogLog Counting相比于LC更加节省内存，空间复杂度只有O(log2(log2(Nmax)))
> - **_HyperLogLog Counting(HLL)_**：HyperLogLog Counting是基于LLC的优化和改进，在同样空间复杂度情况下，能够比LLC的基数估计误差更小

## 原理
hyperloglog的一些发展历程，从比较简单的估算方法开始

### 概率计数
- 假设存在一个hash函数，它对任意输入的输出是在[0,n]这个范围内是随机输出的（这里应该有一定的技巧来控制这个输出的范围）
- 将hash的结果转换成二进制，从低位开始数有多少个0，因为hahs的输出是完全随机的，所以可以认为每一个bit出现0或1的概率都是1/2，所以连续出现k个0的概率就是1/2^k
- 可以计算出所有的hash结果的最长的0的个数R，将2^R作为set的基数

这个计算方式的缺点
1. 估算的基数只能是2的幂，误差会很大
2. 精度不高，如果恰好有一个多个0的hash值，会计算出远超真实结果的基数。也就是算法本生无法处理对中心值的偏离，所以在样本数不够的情况下，这种算法的估算结果会很差

优点：对存储的要求极低，只需要存储最长的0的个数，也就是上面的R值就可以

### LogLog
- 思路比较简单，如果单个概率预测的误差很大，就引入多个，并计算平均值
- 使用多个hash函数，计算出多组的R

最后的基数 = 2^avg(Rx)

缺点：
1. 多组hash的计算引入和计算上的开销
2. 一个改进的方式是进行单次的hash，但是对结果进行分离，存放在多组bucket中

具体的分离方法（stochastic averaging）
![[Pasted image 20220629115138.png]]
- 取hash结果的前k位作为bucket index
- 示例中使用前4为作为index，所以对应16个bucket
- 将除去前k位的后面的结果作为计算值，算出最长0，并且和对应bucket的值进行比较，bucket中始终存放最大值

这种m个bucket的效果可以近似等价于m个hash函数的效果

对于m个bucket，这种方式会将标准差降低1.3/(m^0.5)

> with 2,048 buckets where each bucket is 5 bits (which can record a maximum of 32 consecutive 0s), we can expect an average error of about 2.8 percent; 5 bits per bucket is enough to estimate cardinalities up to 2^27per the original paper and requires only 2048 * 5 = 1.2 KB of memory

### hyper log log
SuperLogLog算法
1. 在计算平均值之前，排除掉最大的值，从而减小偶发性导致的偏差
2. 通过只取最小的70%的数据去计算平均值，标准差从1.3/(m^0.5)降低到1.05/(m^0.5)

hyperloglog
1. loglog中计算平均的方法是几何平均，也就是 avg = (x1*x2*x3)^(1/3)
2. hyperloglog使用调和平均数（harmonic mean）来计算平均值，将方差降低到1.04(m^0.5)hyperloglog也使用superloglog的排除大值的fang'fahyperloglog的me'r


## hyperloglog的merge
1. 两个hyperloglog的规格保持一致（hash函数，bucket size）
2. 比较对应的bucket的值，取较大的那个值

这个特性使得hll非常容易拓展和并发，可以轻松的实现类似map reduce的操作
