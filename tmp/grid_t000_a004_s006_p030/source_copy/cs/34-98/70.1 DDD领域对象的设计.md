## 概览
领域对象包含：
1. 实体 entity
2. 值对象 value object
3. 聚合根 aggregate

一些独立的概念
1. service
2. module/package

## 领域对象设计的原则
### 控制关联
大量的多对多关联对导致实现和维护的困难
1. 规定一个遍历的方向，不要存在方向路径
	- 例如class student中尽量不要包含student的关联，关系是单向的
2. 添加一个限定符，减少多对多的关联
	- 一个student有多个teacher，但只有一个当前在读teacher
3. 消除不必要的关联


### entity和value object
根本的区别：
1. entity：用来表示某种具有==连续性==和==标识==的事物
2. vo：描述某种状态和属性

#### entity
1. 对象的属性可能发生变化，但是有一个贯穿始终的标识可以代表它的身份
2. entity的核心在于它如何被查找、识别、匹配，而非属性或者行为
3. 属性和行为应该被转移到于核心entity关联的其它entity和vo中

==entity应该具有一个全局唯一的id，并在其生命周期中保持不变==
实际操作中，entity应该有一个对应的repo，用来通过id进行操作

#### vo
我们只关心一个模型的属性时，它应该被归位vo

vo在物理存储中可能是和entity放在一行记录中，实现[[反范式化]]

### service
1. service包含一组操作，这些操作不属于任何的对象
2. entity和vo对应的应该是一种事物，service对应的是一种活动
3. service是无状态的

==service的无状态并非是没有副作用，只是service自己的内部是无状态的==，可以简单的类比于网关，网关本身无状态，但是通过网关的操作不是

#### service的粒度控制
当领域对象比较小而关联多时，领域逻辑可能会被泄露到应用层，service相当于提供了一层封装，利用这个封装把领域逻辑控制在领域层中，只向外暴露一个大的service

### aggregate
aggregate是一组相关对象的集合，是数据修改的单元
包含：
1. 一个root
2. 一个boundary

1. 外部对象只能引用root，boundary内部的对象可以相互引用
2. entity在具有全局标识的同时可能存在一个boundary内部的本地标识
3. 当通过root获取内部对象时都只能是值拷贝，也就是外部不能直接修改内部的对象
4. 内部的对象可以持有外部aggregate的root

综述，==aggregate内部对外部是透明的==，只能通过root的方法来进行读取和修改，但这种透明是单向的，内部可以引用外部。
==root的操作可以理解为事务==，需要保证内部的所有对象操作的一致性

每个root都有一个repo，aggregate只能从repo加载,或者由factory整体生产
### module
module是一个内聚的概念集合，不同的module之间是低耦合的
