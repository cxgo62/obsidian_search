# 什么时候需要VO

其实entity是一个非常容易理解的概念，我们的业务中都存在大量的使用全局唯一标识来确定的对象，数据库中一般会有unique key，这种标记会在对象的整个生命周期中都不会发生变化（例如订单号），非常符合连续性的定义。

  

反而是vo作为一种对象是难以被理解的。比如书中说的address这个例子，一般的项目开发中，address仅仅是数据库的一个字段，是一个类的内部变量，感觉完全没有必要作为一个单独的对象来理解。

vo的存在的前提有几点：

1.  信息不是由单个字段构成的，是由多个字段的组合表达出的一种概念
    
2.  这种信息不能通过某个id去查询，常见的情况是以反范式化的形式和某个entity存储在一起
    
3.  这个vo有一些充血方法，不是简单的数据容器
    

  

举个例子，收件人信息这个场景，如果这个收件人信息是作为个人信息记录的几个字段进行存储的，建模时它可能被设定为一个vo。

  

# 什么是service

最简单直观的映象

1.  entity、vo是对象，是一个名词
    
2.  service是一种活动，是一个动词，并且无法归为某个对象的方法。
    

  

service很多时候都是一个比较迷惑的概念，因为是否无法被归到entity、vo是比较难被判断的。举个例子，如果希望给用户发消息，是有个一个msgService更好，还是user对象有一个notify方法更好？

eric说需要判断一下这个动作是不是entity/vo的自然职责，这个也比较模糊。个人认为应该看一下这个功能是否是一种同时被多个场景使用的泛化能力。

上面的例子中，如果全局只有user会有发消息的这种动作，那可能使用notify方法是更合适的，如果存在多重发消息的场景，可能将这种能力抽象成service就更合理，因为在service中可以对全局的消息发送做收口，更容易复用和管理。

  

## application和domain层的service

  

一个service被归属到领域层有两种情况：

1.  一个无状态的动作中包含了很多的业务逻辑，比如msgService中包含了许多消息构建的细节信息
    
2.  service是对多个领域对象操作的封装。这个看起来是应用层干的事情，但是因为这种封装、组合逻辑本身很复杂，且也是业务规则的一部分，为了防止这些业务逻辑泄露到应用层，将原本应该在应用层完成的动作进行下移。领域层的factory也是同样的意思。书中的『资金转账』对应的这种情况。
    

  

所以同理，如果一个service被归属到应用层，那它一定不会包含太多的业务逻辑，这是应用层和领域层划分的一个黄金标准，至于某个部分是否属于业务逻辑，责有很大的主观判定空间。

  

我们一般在非DDD语义下谈论的service，更多的是属于基建层的service，作为一种基础能力提供基础服务，这也更贴合微服务中的service场景。

## 无状态

service的无状态是一个很容易被误解的概念，如果service无状态是否它只能做无副作用的读？

但service的无状态其实指的是service对象本生的无状态，并不是说不会影响全局的其它状态。

几个典型的无状态service：

1.  proxy
    
2.  消息队列的client（不考虑缓存等复杂策略）
    

  

对这些service的调用不会影响service对象内部的状态，但是除非是无副作用的读，否着全局状态一定会被改变
