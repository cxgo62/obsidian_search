## MVCC
1. Multiversion Concurrency Control：多版本并发控制
2. select操作可以不加锁来实现不同的[[32.4 隔离等级]]

## 实现
### 隐藏列
每行记录都添加两个额外的隐藏列：
1. DATA_TRX_ID：最近更新这条记录的事务id
2. DATA_ROLL_PTR：指向回滚行的指针
未指定主键会生成DB_ROW_ID作为标识

### undo log链
undo log的基础概念[[95 WAL 预写日志#undo log]]

- A和B两个事务以如下时序进行操作
- A的事务ID为100
![[Pasted image 20211011103445.png|500]]

A对记录更新完成后，产生一行新的记录，并将DATA_ROLL_PTR指向旧的记录
![[Pasted image 20211011111337.png|600]]

具体的操作过程
1. 加行排他锁，保护整过过程
2. 复制记录，并修改目标字段、DATA_TRX_ID、DATA_ROLL_PTR
3. 将新增行的数据放入到redo log

- delete时过程类似，相当于在顶部加一条软删除标记记录（delete_flag为true），commit时进行物理删除，DATA_TRX_ID的判断标准和select时一致
- insert时不涉及undo log链

### read view
read view实现方式的不同决定了不同的隔离等级

1. read view本质是当前系统中所有活跃的事务的id（为提交的事务）
2. 每个事务有一个独立的read view
3. read view生成的时机决定了RC或RR

read view的结构
1. m_ids，当前活跃（未提交）的事务id
2. up_limit_id，m_ids中最小的id
3. low_limit_id，m_ids中最大的id

一个记录是否可以被读取的判断：
1. 如果DATA_TRX_ID < up_limit_id，历史提交数据，可以读取
2. 如果DATA_TRX_ID > low_limit_id，read view生成之后创建的新事务，不能读取
3. DATA_TRX_ID在up_limit_id和low_limit_id之间，如果在m_ids中，代表未提交，不能读取，不在代表已提交，可以读取


#### RU，serialize
这两种隔离等级不需要使用到mvcc
#### RR
1. 在第一次select时建立read view，后续所有的select都复用这个read view，不再变更
2. 其它写操作和这个read view没有关系
3. 如果发现记录的DATA_TRX_ID存在于自己的read view中，说明这个事务依然活跃（未提交），那就继续顺着undolog list往上找

![[Pasted image 20211011114208.png|500]]
这个例子中，A永远无法读取到B的修改，即使B已经提交，因为A生成read view时B是活跃的

只有在这种时序提交下，B提交后，A才首次进行select，A可以读取到B的数据
![[Pasted image 20211011114443.png|500]]

#### RC
RC和RR的区别就是每次select都重新生成一个read view，这样如果在A运行的过程中B已经提交，A下次select产生的read view就不再包含B，所以可以读取到B的修改

## MVCC的缺陷
![[Pasted image 20211011140416.png]]
第二次select无法读取到B的update，因为B的事务号大于A，会导致发生判断错误

隔离等级只针对select，update等操作无效
https://blog.csdn.net/xmtblog/article/details/105154008/

>  The snapshot of the database state applies to SELECT statements within a transaction, not necessarily to DML statements. If you insert or modify some rows and then commit that transaction, a DELETE or UPDATE statement issued from another concurrent REPEATABLE READ transaction could affect those just-committed rows, even though the session could not query them. If a transaction does update or delete rows committed by a different transaction, those changes do become visible to the current transaction.
