
# CPU使用率突增

## 1 consumer启动导致的cpu节流
consumer启动时，由于reblance的原因，partition的消费会暂停，reblance完成后，会开始处理挤压的消息，导致初期的消费量会变高，触发cpu节流。
现象就是容器cpu被打满，处理延迟上升

> cpu节流：如果并发消费线程数配置过大（目前好多100以上甚至500的），叠加刚启动时消息数量变多，消息消费线程会瞬间创建到配置的最大值如100。在没有配置任何亲和策略的机器上，这些线程会同时运行在物理机的所有核心上，由于容器本身配置只有8核，基于时间片的调度策略，在每个物理核心上线程会限制每秒只能运行8/100秒，产生严重的节流。节流发生后，cpu核心上的线程会频繁切换，对性能有严重的负面影响。

[# 容器环境中避免 CPU 节流](https://blog.crazytaxii.com/posts/avoiding_cpu_throttling_in_a_containerized_environment/)


解决方式：
1. 降低线程数配置
2. 设置subnuma亲和，使得线程被调度在一个subnuma的cpu核心上
3. 使用协程
4.  使用jwarmup进行jit编译，通过预热降低影响

## 3 内存跨numa
cpu和内存在两个numa上，当cpu调用THP大页，会发生跨numa访问，导致严重的性能问题
> 透明大页（Transparent Huge Pages，THP的工作原理
> 通常，操作系统的内存分页是以4KB大小的小页为单位进行管理的。在大规模的内存使用场景下，如数据库、虚拟化环境或大数据处理，频繁的内存页表管理（页表的查找、更新等）会占用大量 CPU 资源。THP 自动将多个连续的 4KB 小页合并为一个 2MB 的大页，从而减少页表项的数量，降低 TLB（Translation Lookaside Buffer）缓存失效的概率，提高内存访问的效率。

解决：容器云问题，重启容器解决
